## AI Automotive Cybersecurity Incident & Compliance Assistant
### Portfolio Prototype Demo

> Speaking pace: ~130-140 words/minute. Estimated runtime: 4:30-5:00.

## 0:00-0:30 — Introduction

What you're looking at is a prototype of an AI Automotive Cybersecurity Incident and Compliance Assistant — a database-backed Python tool designed to organize, query, and eventually analyze cybersecurity incidents across vehicle platforms, ECU zones, connectivity paths, evidence records, mitigation actions, and compliance controls.

The core problem this tool addresses is one real automotive security teams face: incident data can become scattered across logs, VIN records, TARA notes, diagnostic reports, mitigation owners, and compliance references. This project builds the structured data foundation needed to bring that information together.

## 0:30-1:15 — Project Structure

Here is the project folder.

`schema.sql` defines the database tables, constraints, relationships, and the generated risk score field. `seed.py` builds the database and loads realistic automotive cybersecurity seed data. The `data/` folder holds the local SQLite database, `project.db`.

The main file for the backend is `db.py`. This is the database access layer, which keeps SQL and database logic separate from the user interface.

The `app.py` file is a Streamlit front-end preview. It already imports functions from `db.py`, which shows how the backend can support a browser-based analyst dashboard.

## 1:15-2:10 — Database Design

Let me open `schema.sql`.

The schema stores vehicle assets, incidents, security controls, incident-to-control mappings, risk assessments, evidence, and mitigation actions.

The `vehicles` table stores VIN, make, model, model year, ECU zone, and connectivity type. The `incidents` table stores the incident title, description, attack vector, affected component, severity, status, and detection date.

The `risk_assessments` table includes likelihood and impact, and the `risk_score` is generated by SQLite as likelihood multiplied by impact. That is important because the calculation lives in the database, not only in Python, so the score stays consistent regardless of how the data is accessed.

## 2:10-3:10 — Database Access Layer

Now I am opening `db.py`.

At the top, the database path is resolved with Python's `pathlib`, which makes the project easier to move or clone. The `get_connection()` function opens the SQLite database and enables foreign key enforcement.

The core access functions return pandas DataFrames. That matters because Streamlit can display DataFrames directly.

`get_all_incidents()` retrieves incident records. `get_incidents_by_severity()` uses a parameterized query with a `?` placeholder, which is safer than placing user input directly into a SQL string. `get_incidents_with_assets()` uses a JOIN to connect incidents to VINs, vehicle platforms, ECU zones, and connectivity types.

`get_incident_counts_by_severity()` uses GROUP BY and AVG to summarize incidents by severity and risk score. `get_open_mitigations()` returns unresolved actions. `get_incident_evidence_for_ai()` retrieves the database evidence package that a future AI feature could summarize.

## 3:10-4:00 — Terminal Demo

Now I will switch to the terminal.

First I run `python seed.py`. This rebuilds the database from `schema.sql` and loads the seed data.

Next I run `python db.py`. This tests the backend access layer directly.

The output shows all incidents, the parameterized high-severity query, the JOIN query with vehicle assets, the GROUP BY severity summary, open mitigations, and the AI evidence query.

This confirms that Python is connecting to SQLite, executing SQL safely, and returning structured data that can be reused by the application.

## 4:00-4:45 — Front-End Preview and Industry Value

Now I can run `python -m streamlit run app.py` to show the front-end preview.

The dashboard displays incident overview metrics, incident records, compliance control mapping, and a database-backed AI context preview for a selected incident.

The reason this matters is that AI summarization, compliance dashboards, mitigation tracking, and TARA reporting are only useful if the underlying data is structured and traceable. This prototype demonstrates the architecture: a relational schema, secure Python database access functions, a Streamlit front-end preview, and a clear path toward AI-assisted incident review.

This is an early-stage prototype, but the architecture is real, the data model is production-relevant, and the path from here to a deployable tool is clear.
